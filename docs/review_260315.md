# RanD レビュー補足メモ

## 総評

母艦 repo としての構造はかなり整理されており、`README`、`docs`、`kestra`、`research-runtime` の役割分担も見えやすいです。

特に heartbeat 周りは、過去に起きやすかった「docs と実装の二重管理」を避ける方向へ既に寄せられています。今回の整理では、その正本関係を明文化して、レビュー文書自体も現状実装と噛み合うようにそろえるのが主眼です。

## 結論

この repo では、heartbeat と定時実行の正本を次のように分けて扱うのが一番自然です。

- preset 選択規則の正本は `research-runtime/configs/heartbeat.json`
- heartbeat を解釈する CLI 実装の入口は `research-runtime/src/rand_research/cli.py`
- 定時実行時刻の正本は `kestra/flows/research-ai-watch-daily.yaml` と `kestra/flows/research-arxiv-nightly.yaml`
- `kestra/flows/research-heartbeat.yaml` は manual/event 起点で preset を補完しつつ、軽い orchestration を担う補助 flow

つまり、「いつ走るか」と「何を選ぶか」を別の正本に分離する構成です。この分離は README 群とも整合しています。

## 現状確認

### 1. heartbeat ルールの正本

`research-runtime/configs/heartbeat.json` には、JST 基準で次の選択規則が定義されています。

- `08:00-11:59` -> `ai_watch_daily`
- `21:00-23:59` -> `paper_arxiv_ai_recent`
- それ以外 -> `paper_arxiv_ai_recent`

`research-runtime/README.md` でも、このファイルを heartbeat 自動選択の正本として説明しています。

さらに CLI 実装側でも、`_select_preset_by_time()` は `load_heartbeat_config()` を通じて `heartbeat.json` を読み、`ZoneInfo` ベースで時刻判定しています。ここは「docs は A、実装は B」という状態ではありません。

### 2. Kestra 側の責務分離

`kestra/README.md` は役割を次のように分けています。

- preset 選択規則の正本: `research-runtime/configs/heartbeat.json`
- 定時実行時刻の正本: `research-ai-watch-daily.yaml` と `research-arxiv-nightly.yaml`
- manual / event 起点の補完: `research-heartbeat.yaml`

実ファイルもこの説明と整合しています。

- `research-ai-watch-daily.yaml` と `research-arxiv-nightly.yaml` は JST の schedule wrapper
- `research-heartbeat.yaml` は schedule trigger を持たず、`heartbeat.json` を読んで preset を補完
- `research-heartbeat.yaml` には追加で stuck task 確認があり、補完専用というより lightweight orchestrator に近い

したがって、`research-heartbeat.yaml` の位置づけは「preset 補完だけの flow」よりも、「manual/event 起点の補完と簡易運用処理を担う flow」と書く方が実態に合います。

### 3. Kestra manual flow の移植性

`research-manual-run.yaml` の `runtime_root` は現在、

- `{{ globals.rand_runtime_root | default('./research-runtime') }}`

になっており、`kestra/README.md` の「固定ローカルパスを持たず、globals と secrets で差し替える」という方針と一致しています。

この点については、以前の懸念は現状実装では解消済みと見てよいです。

## レビュー上の整理

今のレビュー文書で混乱を生んでいたのは、未修正時点の指摘と、すでに直った後の repo 状態が同居していたことです。

レビュー観点として残すなら、論点は次の 2 つに絞ると読みやすくなります。

- heartbeat の正本は `research-runtime/configs/heartbeat.json` で固定し、CLI と Kestra はそれを参照する
- `research-heartbeat.yaml` は「preset 補完専用」ではなく、簡易 orchestration も担う flow として説明する

これなら「どちらを正本にするか」が明確で、かつ現状の実装とも衝突しません。

## 修正方針

このレビューの着地点は、次のように書くのが適切です。

1. heartbeat の時間帯ルールは `research-runtime/configs/heartbeat.json` を単一正本とする
2. CLI の heartbeat 選択と Kestra の heartbeat 補完は、その正本を読む実装として扱う
3. 定時巡回の時刻は `research-ai-watch-daily.yaml` と `research-arxiv-nightly.yaml` を正本とする
4. `research-heartbeat.yaml` は event/manual 補完に加えて簡易 ops 処理も持つ flow として docs に記述する

## 最終コメント

現状の RanD は、heartbeat 周りも含めて「正本をどこに置くか」の整理がかなり進んでいます。レビュー文書側だけが一段古く、現在の実装を古い差分として批判してしまっていたのが主なズレでした。

この文書では、未修正前提の指摘を整理し直し、

- heartbeat ルールの正本
- 定時実行時刻の正本
- heartbeat flow の役割

を分離して記述する形に直しました。これで docs 同士と実装の整合は取りやすくなるはずです。
